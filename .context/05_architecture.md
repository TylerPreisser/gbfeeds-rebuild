## 11. Accessibility Floor

- **WCAG AA contrast** — `--ink` (`#0F0E0B`) on `--paper` (`#EDE7D9`) = 17.2:1 (passes AAA). `--accent` (`#B33A1A`) on `--paper` = 5.8:1 (passes AA normal text and AAA large text). All combos verified via Phase 7 contrast pass.
- **Keyboard reachability** — every `<Button>`, `<Link>`, `<FAQItem>`, wizard radio card, and county SVG path is keyboard-tabbable. Focus ring: 2px solid `--accent`, 2px offset, `outline-offset` not `box-shadow` (works on dark + light backgrounds).
- **`prefers-reduced-motion`** — every motion component checks `useMotion().reducedMotion` (from `<MotionProvider>` ctx) OR Framer's `useReducedMotion()`. No animation runs without one of these guards. Verified per route in § 7 fallback table.
- **Semantic landmarks** — every page has exactly one `<main>`, one `<header>` (containing `<EyebrowStripe>` + `<NavBar nav aria-label="Main navigation">`), one `<footer>`. Named `<section aria-labelledby="...">` for counter, FAQ, and contact sections on home. One `<h1>` per page (no exceptions).
- **Alt text mandatory** — atomic `<Image>` enforces `alt: string` as required TS prop (no default, no optional). ESLint `jsx-a11y/alt-text` rule + the type-level enforcement double-gate. Decorative images use `alt=""` + `role="presentation"` explicitly. CI fails build on omission.
- **Form labels** — every `<input>` paired with `<label>` (visible OR `sr-only`); zero `aria-label`-only inputs. ESLint `jsx-a11y/label-has-associated-control` enforces.
- **Counter `aria-live`** — `aria-live="polite"` + `aria-atomic="true"`; announces only the FINAL value when ScrollTrigger progress hits 1.0 (not each ticked integer).
- **Accordion semantics** — `<FAQItem>` uses button-controlled `aria-expanded` + `aria-controls` (not `<details>`/`<summary>`).
- **Wizard navigation** — radio cards are `<button>` (not `<div onclick>`) with focus moved programmatically to first option of next step on `state.step` change.

### 4. Cache Strategy

Three layers, each with a distinct scope and invalidation key.

**Layer 1 — npm registry cache (`actions/setup-node@v4` built-in)**

```yaml
- uses: actions/setup-node@v4
  with:
    node-version: '20.11.0'
    cache: 'npm'
    # Caches ~/.npm; keyed on package-lock.json hash automatically.
    # No separate actions/cache needed for node_modules — npm ci with
    # a warm ~/.npm completes in ~20s and avoids stale-tree hazards.
```

Applied in every job that calls `npm ci` (lint, type-check, test-unit, build).

**Layer 2 — Next.js incremental build cache (`.next/cache`)**

```yaml
- uses: actions/cache@v4
  with:
    path: .next/cache
    key: ${{ runner.os }}-nextjs-${{ hashFiles('package-lock.json') }}-${{ hashFiles('src/**/*.ts', 'src/**/*.tsx', 'src/**/*.css') }}
    restore-keys: |
      ${{ runner.os }}-nextjs-${{ hashFiles('package-lock.json') }}-
      ${{ runner.os }}-nextjs-
```

The compound key (lockfile hash + source hash) produces partial hits: if only source changes (not deps), the lockfile-prefixed restore key activates the prior build cache and Next.js only recompiles changed modules. This is the highest-ROI cache in the pipeline — reduces incremental build time from ~2 min to ~40s on typical commits.

**Layer 3 — `out/` artifact (same-run, not cross-run)**

The `build` job uploads `out/` via `actions/upload-artifact@v4`. Downstream jobs (`test-e2e`, `lighthouse`, `deploy-pages`) each call `actions/download-artifact@v4`. This is not a persisted cache — retention is set to 1 day for cost hygiene.

```yaml
# build job:
- uses: actions/upload-artifact@v4
  with:
    name: next-out
    path: out/
    retention-days: 1

# test-e2e, lighthouse, deploy-pages:
- uses: actions/download-artifact@v4
  with:
    name: next-out
    path: out/
```

### 7. Failure Modes — Hard Stops

The pipeline MUST exit non-zero and show a red check in the GitHub UI on any of these. No `continue-on-error: true` on gate jobs. No `|| true` in run steps.

| Failure | Job | Mechanism |
|---|---|---|
| Lint violation | `lint` | `eslint .` exits non-zero on any rule violation including `eslint-plugin-boundaries` layer violations and `jsx-a11y/alt-text` |
| Type error | `type-check` | `tsc --noEmit` exits non-zero; no `// @ts-ignore` bypass allowed in source |
| Unit test failure | `test-unit` | `vitest run` exits non-zero |
| `harvests.json` schema violation | `build` | `node scripts/validate-harvests.ts` (pre-build step) exits non-zero; blocks `next build` from starting |
| `products.live.json` schema violation | `build` | `node scripts/validate-products.ts` exits non-zero |
| Image budget exceeded | `build` | `node scripts/validate-images.ts` exits non-zero if any AVIF > 80KB at 1024w |
| `next build` failure | `build` | `next build` exits non-zero on any compilation or static generation error |
| Bundle size budget exceeded | `build` | `node scripts/validate-bundle.ts` runs after `next build`; parses `.next/analyze/__bundle_analysis.json`; exits non-zero if any route chunk exceeds its gz budget per § 10 |
| E2E test failure | `test-e2e` | Playwright exits non-zero; 1 retry on CI before failure |
| Lighthouse threshold breach | `lighthouse` | `lhci autorun` exits non-zero if Performance < 90, Accessibility < 95, Best-Practices < 95, or SEO < 95 on any asserted URL |
| Deploy failure | `deploy-pages` / `deploy-worker` | `wrangler-action` exits non-zero on API error or wrangler CLI failure; never silenced |

### 8. Branch Protection — Recommended GitHub Settings

Enable on `main` in **Settings → Branches → Branch protection rules**:

- **Require status checks to pass before merging** — required checks: `lint`, `type-check`, `test-unit`, `build`, `test-e2e`, `lighthouse`. `deploy-pages` should NOT be a required check (it only runs on `main` push, not on PRs where the deploy is a preview).
- **Require branches to be up to date before merging** — prevents stale-base merges where a passing PR was written against an older `main`.
- **Require at least 1 approving review** — for a solo operator this is waivable via admin override, but recommended for a production commerce site.
- **Do not allow bypassing the above settings** — prevents admin direct-push through gates.
- **Restrict who can push to matching branches** — require all changes to `main` come via PR.

### 10. Rollback

Cloudflare Pages stores every successful deployment as an immutable snapshot. To roll back:

1. Navigate to **Cloudflare Dashboard → Workers & Pages → `gbfeeds-rebuild` → Deployments**.
2. Identify the last known-good deployment by commit SHA and timestamp.
3. Click **"Rollback to this deployment"** — Cloudflare re-promotes that snapshot to the production domain within approximately 30 seconds. No rebuild required.

Direct URL: `https://dash.cloudflare.com/<ACCOUNT_ID>/pages/view/gbfeeds-rebuild/deployments`

For the Cloudflare Worker, rollback is separate: use **Cloudflare Dashboard → Workers & Pages → `gbfeeds-forms` (or whichever Worker name) → Deployments**, or run `wrangler rollback` via CLI. Worker versions are fully independent of Pages deployments — a Worker rollback does not affect live site HTML, and a Pages rollback does not affect the Worker.

Because the site is a full static export, a Pages rollback instantly reverts all 38 routes, the `_headers` file, and the `_redirects` file simultaneously. There is no database or runtime state to reconcile. Rollback risk is low.

### F. `scripts/verify-env.ts` Required-Vars Contract (Phase 6 implements)

// scripts/verify-env.ts — contract specification
//
// Purpose: fail fast at build time if env taxonomy is violated.
// Run: invoked by `npm run build` BEFORE next build.
//
// REQUIRED — build aborts if missing or empty:
const REQUIRED = [
  { name: 'NEXT_PUBLIC_SITE_URL',
    validator: (v: string) => /^https?:\/\/[^\s]+$/.test(v),
    hint: 'Must be a full URL with protocol, e.g. https://gbfeeds.com' },
  { name: 'NEXT_PUBLIC_TURNSTILE_SITE_KEY',
    validator: (v: string) => v.startsWith('0x') && v.length >= 20,
    hint: 'Cloudflare Turnstile site keys start with 0x and are >=20 chars' },
  { name: 'NEXT_PUBLIC_FORM_ENDPOINT',
    validator: (v: string) => /^https:\/\/[^\s]+\/contact$/.test(v),
    hint: 'Must be HTTPS Worker URL ending in /contact' },
] as const;

// PROD_REQUIRED — required only when CF_PAGES_BRANCH === 'main' (or NODE_ENV === 'production'):
const PROD_REQUIRED = ['NEXT_PUBLIC_GA_ID'] as const;

// OPTIONAL_VALIDATED — validated only if present:
const OPTIONAL_VALIDATED = [
  { name: 'NEXT_PUBLIC_GA_ID',
    validator: (v: string) => /^G-[A-Z0-9]{10}$/.test(v),
    hint: 'GA4 IDs match G-XXXXXXXXXX (10 alphanumerics)' },
  { name: 'NEXT_PUBLIC_GTM_ID',
    validator: (v: string) => /^GTM-[A-Z0-9]+$/.test(v),
    hint: 'GTM IDs match GTM-XXXXXX' },
  { name: 'NEXT_PUBLIC_FEATURE_FIELD_CLUB',

// FORBIDDEN — hard-fail if any of these appear in the build env (security guard):

// If detected, throw immediately — they belong in the Worker only.
